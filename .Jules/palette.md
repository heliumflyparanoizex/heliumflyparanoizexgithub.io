## 2024-05-24 - Focus Visibility using Utility Classes
**Learning:** Adding accessible focus states to interactive UI elements that lack them is critical for keyboard navigation. Using existing Tailwind utility classes (e.g., `focus-visible:ring-2`, `focus-visible:outline-none`, `focus-visible:ring-{color}`) provides an immediate, semantic way to introduce focus visibility without creating custom CSS overrides, keeping the styling tightly coupled to the component architecture.
**Action:** When working in codebases that utilize Tailwind CSS, always prioritize `focus-visible` utility classes for accessible focus styling on interactive elements (buttons, links) rather than relying on custom CSS styles.

## 2026-05-11 - Robust Focus Management for Dynamic Modals
**Learning:** For dynamic modals like lightboxes generated via JavaScript, keyboard accessibility requires more than just adding `tabIndex` and visual focus styles. It's crucial to implement a robust focus management pattern: saving a reference to the element that triggered the modal (`lastFocusedElement`), explicitly shifting focus to an interactive element within the modal (like the close button) when it opens, and restoring focus back to the trigger element when the modal is closed via any interaction (Escape, Enter, click). Without a `setTimeout` when shifting focus to the modal's internal elements, focus may not land correctly due to display/layout timing issues.
**Action:** Always implement explicit focus trapping/shifting and restoration when building or updating custom modals or lightboxes. Ensure trigger elements are accessible via keyboard (e.g., using `role="button"`, `tabIndex="0"`, and `keydown` event listeners for Enter/Space).
