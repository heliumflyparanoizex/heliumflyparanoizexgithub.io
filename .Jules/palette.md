## 2024-05-24 - Focus Visibility using Utility Classes
**Learning:** Adding accessible focus states to interactive UI elements that lack them is critical for keyboard navigation. Using existing Tailwind utility classes (e.g., `focus-visible:ring-2`, `focus-visible:outline-none`, `focus-visible:ring-{color}`) provides an immediate, semantic way to introduce focus visibility without creating custom CSS overrides, keeping the styling tightly coupled to the component architecture.
**Action:** When working in codebases that utilize Tailwind CSS, always prioritize `focus-visible` utility classes for accessible focus styling on interactive elements (buttons, links) rather than relying on custom CSS styles.
## 2026-05-08 - Accessible Modal Focus Management
**Learning:** Custom modal implementations (like lightboxes) must explicitly manage focus to remain accessible. Relying solely on `display: none` and `display: block` creates a keyboard trap. Focus must be programmatically moved into the modal upon opening (often requiring a slight `setTimeout` to ensure the modal is rendered and focusable), and crucially, focus must be returned to the element that triggered the modal (`lastFocusedElement`) upon closing.
**Action:** Always implement a `lastFocusedElement` tracking pattern for custom modals, and ensure standard keyboard interactions (Escape to close, Enter/Space to trigger elements) are manually bound to the modal controls.
