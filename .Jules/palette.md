## 2024-05-24 - Focus Visibility using Utility Classes
**Learning:** Adding accessible focus states to interactive UI elements that lack them is critical for keyboard navigation. Using existing Tailwind utility classes (e.g., `focus-visible:ring-2`, `focus-visible:outline-none`, `focus-visible:ring-{color}`) provides an immediate, semantic way to introduce focus visibility without creating custom CSS overrides, keeping the styling tightly coupled to the component architecture.
**Action:** When working in codebases that utilize Tailwind CSS, always prioritize `focus-visible` utility classes for accessible focus styling on interactive elements (buttons, links) rather than relying on custom CSS styles.
## 2024-03-07 - Dynamic Gallery Lightbox Focus Management
**Learning:** Dynamically generated items (like gallery images) that function as interactive modals/lightboxes must have programmatic accessibility attributes (`tabindex`, `role`, `aria-label`) added during creation. Furthermore, proper modal UX requires focus management: capturing `document.activeElement` before opening the modal, moving focus inside the modal (e.g., to the close button), and restoring focus to the original element upon closing to prevent screen reader/keyboard navigation loss. Adding Escape key support is also a standard pattern for modal closure.
**Action:** When creating any custom modal or dynamic interactive list, strictly implement a focus capture-and-restore pattern and bind keyboard events (Enter/Space to open, Escape to close) alongside standard click handlers.
