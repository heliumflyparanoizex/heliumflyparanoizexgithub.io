## 2024-05-24 - Focus Visibility using Utility Classes
**Learning:** Adding accessible focus states to interactive UI elements that lack them is critical for keyboard navigation. Using existing Tailwind utility classes (e.g., `focus-visible:ring-2`, `focus-visible:outline-none`, `focus-visible:ring-{color}`) provides an immediate, semantic way to introduce focus visibility without creating custom CSS overrides, keeping the styling tightly coupled to the component architecture.
**Action:** When working in codebases that utilize Tailwind CSS, always prioritize `focus-visible` utility classes for accessible focus styling on interactive elements (buttons, links) rather than relying on custom CSS styles.

## 2024-05-24 - Robust Focus Management in Modals/Lightboxes
**Learning:** When implementing custom modals or lightboxes (like an image gallery), it is critical for accessibility to trap and manage focus correctly. Simply opening a modal and relying on the user's screen reader to find it is insufficient. A robust pattern involves capturing the previously focused element (`document.activeElement`) before opening the modal, programmatically shifting focus into the modal (e.g., to the close button) using a small timeout, and explicitly restoring focus to the original element when the modal is closed. Keyboard interactions (Enter/Space to open, Escape to close) must also be handled.
**Action:** Always implement explicit focus capturing, shifting, and restoration when creating or modifying custom interactive overlay components, ensuring the user's place in the document is preserved.
