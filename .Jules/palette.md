## 2024-05-24 - Focus Visibility using Utility Classes
**Learning:** Adding accessible focus states to interactive UI elements that lack them is critical for keyboard navigation. Using existing Tailwind utility classes (e.g., `focus-visible:ring-2`, `focus-visible:outline-none`, `focus-visible:ring-{color}`) provides an immediate, semantic way to introduce focus visibility without creating custom CSS overrides, keeping the styling tightly coupled to the component architecture.
**Action:** When working in codebases that utilize Tailwind CSS, always prioritize `focus-visible` utility classes for accessible focus styling on interactive elements (buttons, links) rather than relying on custom CSS styles.

## 2025-03-08 - Dynamic Element Accessibility & Modal Focus Restoration
**Learning:** Dynamically injected elements (like gallery images) that are meant to be interactive must have their accessibility attributes (`tabindex="0"`, `role="button"`) and keyboard events (Enter/Space) attached during JavaScript DOM creation, otherwise they are completely invisible to keyboard-only users. Furthermore, opening a modal (like a Lightbox) must shift focus into the modal and perfectly restore focus to the original triggering element (`lastFocusedElement`) upon closing, otherwise the user loses their place in the document flow.
**Action:** When creating custom modals or dynamically generated lists of interactive items, strictly enforce the "Focus Management Pattern" (save original element -> shift focus inside modal -> restore focus on close) and always pair `click` listeners with `keydown` listeners for Enter/Space.
