## 2024-05-24 - Focus Visibility using Utility Classes
**Learning:** Adding accessible focus states to interactive UI elements that lack them is critical for keyboard navigation. Using existing Tailwind utility classes (e.g., `focus-visible:ring-2`, `focus-visible:outline-none`, `focus-visible:ring-{color}`) provides an immediate, semantic way to introduce focus visibility without creating custom CSS overrides, keeping the styling tightly coupled to the component architecture.
**Action:** When working in codebases that utilize Tailwind CSS, always prioritize `focus-visible` utility classes for accessible focus styling on interactive elements (buttons, links) rather than relying on custom CSS styles.

## 2025-05-18 - Interactive Elements and Modal Focus Management
**Learning:** Dynamically generating interactive `div` elements for a gallery lacks intrinsic keyboard support (unlike `<button>`). Additionally, custom modals (like a lightbox) must explicitly manage focus. Failing to shift focus into the modal upon opening and failing to restore it to the triggering element upon closing traps keyboard users and disrupts their navigation flow.
**Action:** When dynamically creating interactive elements without semantic HTML (like a `div` acting as a button), always programmatically apply `tabIndex="0"`, `role="button"`, `aria-label`, and a `keydown` handler for Enter/Space. When implementing custom modals, capture `document.activeElement` on open, `.focus()` an element inside the modal, and restore focus to the captured element on close.
