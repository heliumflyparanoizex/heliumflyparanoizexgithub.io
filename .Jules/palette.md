## 2024-05-24 - Focus Visibility using Utility Classes
**Learning:** Adding accessible focus states to interactive UI elements that lack them is critical for keyboard navigation. Using existing Tailwind utility classes (e.g., `focus-visible:ring-2`, `focus-visible:outline-none`, `focus-visible:ring-{color}`) provides an immediate, semantic way to introduce focus visibility without creating custom CSS overrides, keeping the styling tightly coupled to the component architecture.
**Action:** When working in codebases that utilize Tailwind CSS, always prioritize `focus-visible` utility classes for accessible focus styling on interactive elements (buttons, links) rather than relying on custom CSS styles.

## 2025-03-10 - Keyboard Accessibility & Focus Management for Modals
**Learning:** For purely visual gallery interfaces (lightboxes), making dynamically generated non-semantic elements (`div`s representing images) accessible requires not just `tabindex="0"`, but explicitly mapping `Enter` and `Space` keydowns to click triggers, and ensuring focus trapping. Upon opening the modal, moving focus to the close button (and restoring it back to the original element on close) completes the loop. In custom CSS environments, utilizing `:focus-visible` with a clear outline matching existing CSS variables (e.g., `var(--color-primary)`) ensures the focus ring is clear but only appears for keyboard users.
**Action:** When implementing custom interactive elements (like gallery images) that are not natively focusable/clickable buttons, always inject `tabindex`, keyboard event listeners (`keydown`), and handle explicit focus capture/restoration loops when opening/closing associated modals.
