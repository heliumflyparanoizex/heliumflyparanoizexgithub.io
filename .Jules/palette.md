## 2024-05-24 - Focus Visibility using Utility Classes
**Learning:** Adding accessible focus states to interactive UI elements that lack them is critical for keyboard navigation. Using existing Tailwind utility classes (e.g., `focus-visible:ring-2`, `focus-visible:outline-none`, `focus-visible:ring-{color}`) provides an immediate, semantic way to introduce focus visibility without creating custom CSS overrides, keeping the styling tightly coupled to the component architecture.
**Action:** When working in codebases that utilize Tailwind CSS, always prioritize `focus-visible` utility classes for accessible focus styling on interactive elements (buttons, links) rather than relying on custom CSS styles.

## 2026-04-24 - Focus Visibility styling in non-Tailwind projects
**Learning:** When adding `:focus-visible` utility classes in a project that includes Tailwind CSS via a CDN tag in HTML, but doesn't have a build step for it or uses primarily custom semantic CSS, relying on Tailwind arbitrary classes like `focus-visible:ring-[var(--color-primary)]` might not work or just clutters the HTML if they are completely inert. It's safer and cleaner to add a shared block of CSS in `style.css` targeting the semantic classes directly using the `:focus-visible` pseudo-class (e.g. `.nav-link:focus-visible { outline: 2px solid var(--color-primary); }`).
**Action:** When working in codebases, even if Tailwind is present via CDN, if the project primarily uses custom CSS classes, prefer adding focus visibility styles to the main `style.css` file rather than appending many arbitrary Tailwind utility classes to HTML elements.
