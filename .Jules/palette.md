## 2024-05-24 - Focus Visibility using Utility Classes
**Learning:** Adding accessible focus states to interactive UI elements that lack them is critical for keyboard navigation. Using existing Tailwind utility classes (e.g., `focus-visible:ring-2`, `focus-visible:outline-none`, `focus-visible:ring-{color}`) provides an immediate, semantic way to introduce focus visibility without creating custom CSS overrides, keeping the styling tightly coupled to the component architecture.
**Action:** When working in codebases that utilize Tailwind CSS, always prioritize `focus-visible` utility classes for accessible focus styling on interactive elements (buttons, links) rather than relying on custom CSS styles.

## 2025-05-17 - Accessible PDF Codex Viewer Navigation Controls
**Learning:** In the custom Codex PDF viewer implementation, interactive navigational controls (`.codex-tab`, `.control-btn`, `#pdf-download`) lacked explicit keyboard focus states (`focus-visible` styling), making them difficult for keyboard-only users to navigate. Additionally, the `#pdf-download` action was represented by an icon-only element without an `aria-label`, resulting in an inaccessible experience for screen readers. This represents a pattern in custom-built viewers where typical accessible control patterns are often overlooked compared to standard UI forms.
**Action:** Implemented clear focus indicators by adding standard Tailwind CSS `focus-visible` utility classes to the custom viewer's interactive elements to ensure visibility during keyboard navigation. Added an explicit `aria-label="Download PDF"` to the icon-only download link. In the future, ensure all custom widgets or viewers are verified for keyboard focus visibility and that any icon-only controls receive explicit ARIA labels.
