## 2024-05-24 - Focus Visibility using Utility Classes
**Learning:** Adding accessible focus states to interactive UI elements that lack them is critical for keyboard navigation. Using existing Tailwind utility classes (e.g., `focus-visible:ring-2`, `focus-visible:outline-none`, `focus-visible:ring-{color}`) provides an immediate, semantic way to introduce focus visibility without creating custom CSS overrides, keeping the styling tightly coupled to the component architecture.
**Action:** When working in codebases that utilize Tailwind CSS, always prioritize `focus-visible` utility classes for accessible focus styling on interactive elements (buttons, links) rather than relying on custom CSS styles.

## 2024-05-24 - Robust Focus Management for Custom Modals/Lightboxes
**Learning:** When implementing or enhancing custom modals and lightboxes (like image galleries), managing the user's focus is critical for an accessible keyboard experience. It's not enough to just make items clickable; we must capture the previously focused element before opening, shift focus into the modal (e.g. using `setTimeout` to ensure the modal's close button is focusable), allow closing via the `Escape` key, and reliably restore focus to the original element when closing.
**Action:** For all future custom overlays/modals/lightboxes, implement a focus management flow: store `document.activeElement` on open, shift focus within the modal, and restore focus on close.
